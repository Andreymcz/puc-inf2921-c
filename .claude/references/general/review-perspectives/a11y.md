# A11Y — Accessibility

## Essential

- [P0] Does this meet WCAG AAA target requirements (contrast, keyboard navigation, screen readers)?
- [P0] Are interactive elements focusable with visible focus indicators?
- [P0] Are focus management and live region announcements tested with actual screen readers (NVDA, VoiceOver)?
- [P0] Are all interactive custom components (modals, dropdowns, pickers) fully keyboard-operable with documented shortcuts?
- [P0] Do dynamic content updates (SPAs, AJAX regions, toasts) produce correct screen-reader announcements via aria-live, verified across NVDA, JAWS, and VoiceOver?
- [P0] Can every workflow be completed keyboard-only, with no focus traps, logical tab order, and skip-navigation links?
- [P0] Does the product satisfy applicable legal mandates (ADA, Section 508, EN 301 549, EAA), with a published accessibility statement?

## Deep-dive

- [P1] Are ARIA labels and roles used where semantic HTML is insufficient?
- [P1] Do color-dependent cues provide a non-color alternative, and do palettes pass WCAG AA/AAA contrast ratios?
- [P1] Are instructions and error messages in plain language, with timeouts/animations accommodating cognitive disabilities?
- [P1] Has a formal WCAG 2.2 Level AAA audit been performed, with failures tracked to resolution?
- [P1] Are ARIA roles/states/properties used only when native HTML is insufficient, and do composite widgets follow WAI-ARIA Authoring Practices?
- [P1] Can drag/gesture/hover interactions be performed via alternative single-pointer or switch-accessible controls (target >= 44x44 CSS px)?
- [P2] Are custom widgets built on platform primitives that expose correct accessibility APIs (name, role, value, states)?
- [P2] Are generated PDFs tagged with correct structural markup and verified with accessibility checkers (PAC, Acrobat)?
