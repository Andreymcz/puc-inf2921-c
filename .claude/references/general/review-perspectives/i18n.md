# I18N — Internationalization

## Essential

- [P0] Are all user-facing strings using i18n keys (not hardcoded)?
- [P0] Are both locale files (pt-BR.json, en-US.json) and .po catalogs updated in sync?
- [P0] Are resource bundles structured so new locales add without code changes, with automated extraction/merge pipelines preventing stale or orphaned keys?
- [P0] Are text pipelines fully Unicode-safe (UTF-8, NFC normalization, no mid-codepoint truncation), with grapheme clusters handled for truncation/counting/cursor movement?

## Deep-dive

- [P1] Are diacritics and special characters properly encoded?
- [P1] Are locale fallback chains tested (missing key in primary locale falls back gracefully)?
- [P1] Are date, number, and currency formats locale-aware in both backend and frontend, using locale-sensitive formatters (ICU MessageFormat, Intl APIs) for plurals, ordinals, relative time, units, and number grouping?
- [P1] Does the translation workflow integrate with a TMS (Crowdin, Phrase, Lokalise) via CI (no manual file copying)?
- [P1] Does the UI adapt to RTL (Arabic, Hebrew) with correct layout/icon mirroring and bidi isolation (`dir="auto"` or Unicode bidi controls)?
- [P2] Are culturally sensitive elements (colors, imagery, idioms, name/address order, humor) reviewed for target markets, and is there an automated QA gate (pseudo-localization, string-length expansion, screenshot diffing) catching untranslated/clipped text?
- [P2] Do search indexes and sort collations use locale-aware rules (ICU collation, accent-insensitive matching)?
- [P2] Are internationalized interfaces accessible (correct `lang` per fragment, screen-reader-friendly locale switches, contrast/spacing for Devanagari or CJK)?
- [P3] Is there a content strategy defining which locales receive full/partial/MT coverage, with metadata (completeness %, last-updated) exposed to stakeholders?
