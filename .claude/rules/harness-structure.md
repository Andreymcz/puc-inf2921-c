# Harness Structure

Lean inventory of the SEJA Claude harness components. Auto-loaded when working on `.claude/` files.

## Skills

Canonical lifecycle path (matches `CLAUDE.md § Key workflows`): `/research` (or `/explain`) > `/design` | `/plan` > `/implement` > `/check` > `/document` | `/communicate` > `/reflect` -- validate before you communicate. Skills are inventoried below in alphabetical/grouping order, not lifecycle order.

16 skills in `.claude/skills/<name>/SKILL.md` -- 14 user-facing + 2 internal lifecycle hooks (pre-skill, post-skill). The `/document` skill generates and updates project documentation based on plan Docs: fields or auto-detected changes. The `/pending` skill lists and addresses outstanding human actions from the pending-actions ledger (`_output/pending.jsonl`). Action types include `mark-implemented`, `test-implementation`, `verify-as-coded`, `update-documentation`, `apply-promote-markers`, `periodic-curation`, `spec-drift-check`, `incorporate-research-markers`, `create-decision-entry`, `implement` (plan source: filed at `/plan` completion via post-skill, closed at `/implement` rename -- see plan-000408; roadmap source: filed at `/plan --roadmap` step 10 via `--source roadmap-<id>`, closed at `/implement --roadmap` Phase 2 step 14), and `user-defined`; `PUBLISH:` description-prefixed entries are filed by `tools/cut_tag.py`. Each user-facing skill ships a `SKILL-quickguide.md` sibling file with a designer-friendly description, example, and usage scenario; SKILL.md carries a mandatory blockquote pointer to the sibling (see `harness-governance.md § Sibling SKILL-quickguide.md pattern`). The `/plan` skill supports three modes: standard planning, `--light` (lightweight proposals), and `--roadmap` (full product roadmap). The `/research` skill includes a step 9b decision-extract proposal flow that detects design decisions in advisory Q&A and offers to create structured D-NNN entries via `apply_marker.py --marker DECISION_APPEND`, with options to create now, defer to the pending ledger, or skip. Pre-skill executes as a pipeline of 6 composable stages (help, brief-log, budget-eval, pending-check, ref-load, constitution) -- 3 critical (always run) and 3 non-critical (error-isolated, skippable). `orphan-check` and `compaction-check` are now sub-steps of `budget-eval` (legacy stage IDs remain recognized for backward compatibility with `skip_stages`). Skills can declare `skip_stages: [stage-id, ...]` in their YAML frontmatter (`metadata.skip_stages`) to opt out of non-critical stages.

## Agent-Agnostic References

Shared reference files are organized into three groups. `product-design/` (the directory) holds user-authored project configuration; "context" (the concept) refers to the LLM context window -- these terms are unrelated.

- General references (18 + 16 per-perspective + 6 per-onboarding + 5 per-communication): `general/*.md` files (including `general/threat-model.md` -- harness threat model with STRIDE-lite vectors, trust boundaries, and mitigation status; `general/documentation-quality.md` -- documentation quality standard with Diataxis classification, freshness policy, and structural-completeness rules; `general/designer-copy-voice.md` + `general/designer-copy-voice-rationale.md` -- agent-facing copy voice/encoding rules and their maintainer-only rationale sibling (SKILL-rationale pattern); `general/batch-execution-pattern.md`, `general/coding-standards.md`, `general/constraints.md`, `general/permissions.md`, `general/skill-graph.md`), `general/review-perspectives/` with Essential/Deep-dive tiers (P0-P4) and an auto-generated Essential summary, `general/onboarding/` with role families (BLD/SHP/GRD) and expertise levels (L1-L3: Contributor, Expert, Leader), and `general/communication/` with audience segments (EVL/CLT/USR/ACD).
- Template references (49): `template/*.md`, `template/*.json`, `template/agent/*.yaml`, `template/ci/`, `template/demo/`, and `template/docs/` -- used by `/design` to generate project-specific files. Includes `template/constitution.md` for immutable project principles (copied by `/seja-setup`, instantiated by `/design`), `template/product-design-changelog.md` for the separated conceptual design changelog, `template/agent/` for agent-facing structured specifications (constraints, entities, permissions, spec-checks), `template/demo/` (4 files: pre-filled conventions, constitution, conceptual design, and walkthrough for the `/seja-setup --demo` hello-world experience), and `template/docs/` (6 documentation templates with `freshness` and `diataxis` classification metadata; the contextual-help template documents the expanded 6-section pattern).
- Project-specific references: `project/*.md` and `project/agent/*.yaml` -- generated per-project by `/design`. Includes `project/constitution.md` (immutable principles -- required for new projects; `/check health` validates its presence as Check 7; existing projects without it remain functional), `project/agent/` (agent-facing structured specs for automated validation), design intent (`project/product-design-as-intended.md` -- unified working intent + DRR-shaped Decisions section + CHANGELOG, Human (markers) classification since SEJA 2.8.3; plan-000268 merged the former `design-intent-to-be.md` and `design-intent-established.md` layers into this single file), implementation state (`project/product-design-as-coded.md` -- unified as-coded file with three H2 domain sections: `## Conceptual Design`, `## Metacommunication`, `## Journey Maps`; Agent-maintained via post-skill; section-boundary enforcement via `check_section_boundary_writes.py`; unified by SEJA 2.8.4 / plan-000269 from the former three as-is files), conceptual-design changelog (`project/product-design-changelog.md` -- kept separate from product-design-as-coded.md; Phase 3 F from advisory-000264 will conditionally embed it in a future release), and journey map files: JM-TB-NNN designed journeys live in `project/product-design-as-intended.md §15`; JM-TB-NNN implemented journeys live in `project/product-design-as-coded.md § Journey Maps`; JM-E-NNN discovered journeys live in `project/ux-research-results.md §5` (Discovered User Journeys, append-only for prose; markers via apply_marker.py). The UX research file was consolidated into a single `project/ux-research-results.md` in SEJA 2.8.2 (plan-000267, Human (markers) classification). The As-Intended / As-Coded Registry in `conventions.md` declares all registered as-intended/as-coded file pairs with a Section column (enabling multi-artifact-type files; tools discriminate artifact types by ID prefix -- JM-TB-NNN vs JM-E-NNN) and is the single source of truth for `/design`, `/explain spec-drift`, and post-skill DONE marker proposals. Lifecycle markers (`STATUS: implemented` + legacy `STATUS: IMPLEMENTED`, `ESTABLISHED:`) defined in `general/shared-definitions.md` track the lifecycle of individual as-intended items; the promote workflow (`/explain spec-drift --promote`) is split into Phase 3a (Decision entry proposal generation) and Phase 3b (`--apply-markers plan-<id>`, STATUS marker flip via `apply_marker.py`). Decision entries follow a DRR shape with stable `D-NNN` IDs in a namespace orthogonal to REQ IDs.

## Helper Scripts

76 scripts distributed across `.claude/skills/scripts/` (shared utilities) and co-located skill subdirectories (skill-owned scripts live next to their `SKILL.md`) -- validation checks, index generators, conversion utilities, the central `project_config.py` module that parses `project/conventions.md` to supply all scripts with project-specific paths and settings, `run_all_checks.py` (CI-independent validation orchestrator that discovers and runs all check scripts with unified exit codes and JUnit XML output), `check_docs.py` (plugin-based documentation consistency checker with 6 scanners: harness integrity, path liveness, env vars, command refs, terminology, structural-completeness), `check_git_freshness.py` (compares local branches to their upstreams for the codebase and companion workspace; never pulls; invoked by `/check freshness` and by the `check-git-freshness` periodic trigger), `check_worktree_health.py` (detects orphaned git worktrees from parallel wave execution; invoked by `/check health`), `check_harness_drift.py` (compares a canonical source harness against a target workspace to detect added, removed, or revised files; invoked by `/check health --source`), `apply_marker.py` (sole write path for Human (markers) files, supporting STATUS, ESTABLISHED, INCORPORATED, CHANGELOG_APPEND, and DECISION_APPEND marker types; DECISION_APPEND appends new `### D-NNN:` DRR-shaped decision entries to the `## Decisions` section with auto-assigned IDs), `generate_decision_digest.py` (generates `_output/decision-digest.jsonl` from D-NNN entries in `product-design-as-intended.md` and HIGH/MEDIUM recommendations in advisory logs; invoked conditionally by post-skill step 7d), `backfill_decision_digest.py` (one-time migration utility to extract decisions from existing advisory logs into the decision digest; idempotent with `--dry-run` support), `generate_changelog_data.py` (extracts structured change data from git history for harness changelog generation; used by `/document --type changelog --harness`), and `generate_call_graph.py` (statically parses SKILL.md, agent `.md`, and scripts to discover 7 typed node classes (skill, agent, script, rule, ref-general, ref-template, ref-project) and 6 edge types (invokes, delegates, orchestrates, eager-load, lazy-load, imports); emits `.claude/references/general/call-graph.json` + `seja-public/docs/concepts/call-graph.md` (Mermaid overviews + per-skill drill-downs + reverse-index tables + text fallback) + `seja-public/docs/concepts/call-graph.{html,css,js}` (interactive Cytoscape.js viewer with filter-by-type, three layouts cose/dagre/concentric, manual drag with localStorage, collapsible side panel, SVG/PNG export); `--check` mode validates output freshness + edge-target resolvability, wired into `run_preflight_fast.py` as a convention linter).

## Subagent Prompts

16 subagent prompts in `.claude/agents/`, organized by role:

- **Evaluator agents** (9): review artifacts against quality perspectives -- code-reviewer, council-debate, harness-health-evaluator (9 built-in checks including harness drift detection via `--source`), migration-validator, plan-reviewer, research-reviewer, semiotic-inspector, standards-checker, test-runner.
- **Generator agents** (7): produce self-contained artifacts from well-defined inputs -- communication-generator, onboarding-generator, document-generator, test-plan-generator, explanation-generator (produces explanation reports for behavior/code/data-model modes -- invoked by the `/explain` thin-wrapper for non-interactive analysis modes), architecture-explainer (produces architecture explanation reports -- multi-phase survey/identify/write workflow; invoked by the `/explain` thin-wrapper for the architecture mode), evolution-explainer (produces behavior-evolution explanation reports -- plan-history mining + timeline-building + write workflow; invoked by the `/explain` thin-wrapper for the behavior-evolution mode). Invoked by thin-skill wrappers that handle argument parsing, interactive prompts, and lifecycle hooks. Generator agents must receive the project constitution as part of their prompt for trust boundary enforcement.
- **Executor agents** (pattern): execute plan steps in isolated context windows. Used by `/implement` auto mode. Not standalone prompt files -- the implement skill constructs their prompts dynamically from plan step metadata.
