## 1. Platform Purpose

A local RAG knowledge base for a university course team. Ingests `.md` and `.pdf` documents into ChromaDB and exposes a `query_knowledge` tool via MCP, allowing AI assistants (Copilot, Claude) to retrieve relevant course knowledge during chat sessions.

The platform addresses the problem of scattered, hard-to-retrieve course knowledge across five team members with different backgrounds (engineering, social science, design). By centralizing knowledge in a queryable vector store and injecting retrieval into AI assistant sessions, team members can ask questions in natural language and get answers grounded in the team's own documents.

### Design Philosophy

Knowledge should be accessible in context — where the team is already working (inside an AI assistant), not in a separate tool.

<!-- REQ-ENT-001 -->
### KnowledgeBase

> - The top-level container for all ingested documents
> - Corresponds to the ChromaDB collection
> - Single instance per deployment (single-user tool)

### Document

> - A source file (.md or .pdf) ingested into the knowledge base
> - Identified by file path and checksum
> - Can be re-ingested when updated

### Chunk

> - The atomic unit of retrieval — a text segment from a Document
> - Embedded as a vector in ChromaDB
> - Carries metadata: source file, page/section, ingestion timestamp

## 3. Domain-Specific Concepts

### RAG Pipeline

The Retrieval-Augmented Generation pipeline:
1. **Ingest**: Load documents, split into chunks, embed with sentence-transformers, store in ChromaDB
2. **Query**: Embed the user's question, retrieve top-k similar chunks from ChromaDB, return as context
3. **MCP exposure**: The `query_knowledge` MCP tool wraps the Query step for AI assistants

### MCP Server

A Model Context Protocol server that exposes `query_knowledge` as a tool callable by Claude or Copilot during chat sessions, injecting retrieved chunks as context.

## Decisions

> ADR-shaped decisions. Append-only. Written by human or agent (via apply_marker.py after confirmation).

### D-001: Use ChromaDB as the vector store

- **Status:** proposed
- **Context:** Team needs a local, zero-infrastructure vector store for a small document corpus (~50 documents). Alternatives considered: FAISS (no metadata filtering), Qdrant (needs a server), Weaviate (too heavy).
- **Decision:** ChromaDB — embedded, no server, metadata filtering, Python-native.
- **Consequences:** Single-writer limitation acceptable for a single-user tool. Migration to a server-based store is possible if multi-user support is needed later.
